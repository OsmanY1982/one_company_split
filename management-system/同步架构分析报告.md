## 三、用户权限模型分析

### 3.1 权限体系

`services/permission_service.py` 实现了完整 RBAC：

```
超级管理员 (super_admin)  ←── 所有权限
  └── 管理员 (admin)      ←── 除 system/role 外全部
        └── 经理 (manager)   ←── 查看/编辑/审批
              └── 操作员 (operator) ←── 基础 CRUD
                    └── 查看员 (viewer) ←── 只读
财务 (finance)            ←── 财务专属

权限类型: create / read / update / delete / export / import / approve / transfer / admin
资源类型: order / product / customer / employee / finance / inventory / report / setting / user / role / system
```

### 3.2 管理员角色检查点

| 文件 | 判断逻辑 | 说明 |
|---|---|---|
| `core/app_state.py:45` | `self._role == 'admin'` | 全局 `is_admin` 属性 |
| `core/user_dao.py:168` | `role == "管理员"` | `is_admin()` 方法，使用中文 |
| `core/auth_service.py:15` | `username == "admin"` | `get_user_role()`，按用户名硬编码 |
| `modules/admin/admin_user.py:138` | `role == 'admin'` | 过滤管理员不在用户列表展示 |
| `modules/admin/admin_user.py:291` | `username == 'admin'` | 禁止删除 admin 账号 |
| `config/admin.json` | bcrypt 密码 | 独立配置文件，非数据库 |

### 3.3 权限同步状态

**permissions.db 完全未被任何同步机制覆盖**。角色分配、权限变更仅在本地生效。

## 六、风险与建议

### 6.1 高风险项

1. **密码明文存储**：`admin_user.py` 和 `business_service.py` 使用明文密码字段，需统一迁移到 bcrypt
2. **admin.json 独立于数据库**：管理员密码不应通过文件管理，应纳入 `users.db` 或 `admin.db`
3. **auth_service.py 缺失**：`admin_user.py` 导入不存在的模块，代码在运行时将崩溃

### 6.2 建议执行顺序

1. **先建基础**：创建 `modules/auth/auth_service.py`（统一认证入口）→ 修复 `admin_user.py` 崩溃
2. **再建 UI**：创建 `modules/auth/login_window.py`（登录/注册窗口）
3. **统一数据**：修正 `password` 字段命名，统一 `role` 枚举值
4. **补齐同步**：在 `simple_sync.py` 增加 `sessions` / `permissions` / `admin_config` 表
5. **接入装饰器**：在所有 CRUD 入口部署 `@sync_after` 装饰器
6. **云端建表**：在 Supabase 创建 `sessions` / `permissions` / `roles` / `role_permissions` / `user_roles` / `admin_config` 表
